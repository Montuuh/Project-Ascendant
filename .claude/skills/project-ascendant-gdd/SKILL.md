# Project Ascendant — GDD Workflow Skill

Project Ascendant is a Unity roguelike deckbuilder. The Game Design Document
(GDD) lives in Notion as the single source of truth. This skill governs how
Claude Code interacts with it.

---

## 1. Canonical Documents

Two living Notion documents. Read the page index in
`references/page-index.md` for all URLs.

| Document | Purpose |
| --- | --- |
| **GDD** | 10 topic sub-pages. Section numbers §N.N.N preserved as heading prefixes. |
| **BACKLOG** | Project state, gap log, active debates, sequential changelog. |

**Read precedence:**
1. Notion GDD and BACKLOG = canonical. Always read before implementing.
2. Any local `.md` snapshot in `docs/gdd/` = read-only archive. Never edit it
   directly. It is generated by the export script (see §6).
3. User instruction in chat overrides canonical for the current session only.
   Write the resolution back to Notion before ending the session.

---

## 2. Before Implementing Any System

**Always fetch the relevant GDD topic page before writing code.**

```
1. Read references/page-index.md to find the correct Notion URL.
2. Call notion-fetch on that URL.
3. Locate the §N.N.N section that specifies the system you are implementing.
4. If the section does not exist or is marked PENDING → stop, flag to user,
   do not invent a spec.
```

This is not optional. Silent drift between code and design doc is the
primary failure mode for this project.

---

## 3. Write Authority by Topic Status

Each GDD topic page carries a status in its header.

| Status | What Claude Code may do |
| --- | --- |
| 🔒 **Locked** | Read freely. Write only to add `> ⚠️ OPEN:` flag blocks (see §4). Never rewrite locked prose without explicit user instruction. |
| 🟢 **In Progress** | Read and write freely. Add content, resolve open questions, update sections after user confirms decisions. |
| 🟡 **Pending** | Read freely. Write only scaffolding bullets. No GDD-grade prose. |

**Re-opening a Locked topic** requires the user to say so explicitly. When
unlocked: acknowledge it, note it in the BACKLOG changelog, do the work,
then ask for explicit re-lock confirmation.

---

## 4. Flag-Don't-Resolve Protocol

When implementing a system and you encounter an ambiguity, contradiction, or
gap in the spec:

**Do not resolve it silently in code.**

Instead:
1. Add a flag block to the relevant GDD page using `notion-update-page`
   with `update_content`:

   ```
   > ⚠️ OPEN (Claude Code, <date>): <specific question or contradiction>.
   > Blocked: <what implementation decision is pending this resolution>.
   ```

2. Add a corresponding entry to the BACKLOG gap log (Severity 3 unless it
   blocks implementation, in which case Severity 2).

3. Tell the user what you flagged and why before continuing.

4. Implement a safe default or stub in code, clearly commented:
   ```csharp
   // TODO: Pending GDD clarification — see §N.N.N OPEN flag in Notion.
   // Current stub behaviour: <describe what the stub does>.
   ```

---

## 5. BACKLOG Updates

Update the BACKLOG when:
- A topic status changes (e.g., Pending → In Progress after work starts).
- A gap is resolved (mark ✅ Resolved in the gap log table).
- A new gap is discovered during implementation.
- A design decision is confirmed by the user mid-implementation session.

Always append to the **Sequential Changelog** table at the bottom of the
BACKLOG with the date and a one-line summary. Never delete changelog rows.

---

## 6. GDD Export to Git (run on demand)

When the user asks to "snapshot the GDD", "export to git", or "sync docs":

```bash
# Install once
npm install -g notion-to-md

# Run the export script (create it if it doesn't exist yet)
node docs/scripts/export-gdd.js
```

If `export-gdd.js` does not exist, create it at `docs/scripts/export-gdd.js`
using the template in `references/export-script-template.js`. The script
exports each GDD topic page to `docs/gdd/topic-N-<slug>.md`. After export,
commit with message `docs: snapshot GDD — <topic(s) changed>`.

The exported files are **read-only archives**. Never edit them directly.

---

## 7. Section Numbering Convention

The GDD uses §N.N.N notation for cross-references (e.g., §3.3.5.1, §4.2.3).
These are preserved as heading prefixes in every Notion page.

When referencing the spec in code comments or commit messages, always use
the section number: `// Per §3.3.5.1 — Faint precedence rule`. This makes
grep-based cross-referencing between code and design doc trivial.

## 9. Key Design Constraints (do not violate in code)

These are load-bearing rules from locked GDD sections. If an implementation
decision would violate one, stop and flag it.

- **0 HP = Fainted.** No separate IsFainted flag. `CurrentHP == 0` is the
  fainted state. (§2.4.1)
- **Faint overrides Freeze.** If a Frozen Lead faints, the position-lock is
  void. (§3.3.5.1)
- **Mastery Moves are immutable.** TMs and Tutors cannot replace them. (§4.3.9.2)
- **Backstrike on empty slot fizzles.** Does not redirect to Lead. (§4.3.4.1)
- **Cleave never fizzles.** Hits all non-fainted slots, minimum 1. (§4.3.4.1)
- **Manual swap cost scales per turn:** 1st=1AP, 2nd=2AP, 3rd=3AP. SF/SB
  do NOT increment this counter. (§3.3.1)
- **Stat stages persist across boss phase transitions.** (§4.4.3.1)
- **Champion buff cap:** +5% Attack per defeated ally, max +20%. (§4.7.1)
- **In-combat revival is forbidden** unless consumable explicitly says
  "Revives" (no launch consumable does). (§2.4.3)
- **Consumables restore at combat end.** They are not expendable. (§3.5)
- **All damage/effects resolve on slot, not on Pokémon.** Intents target
  positions. Whoever occupies the slot at Resolution time takes the hit. (§4.3.2)
